All Software Engineers are programmers, but not all programmers are Software Engineers

Admin Software Engineer 15

Normally, when someone knows how to write code that result to an app or website, people refer to that person as software engineer. That's correct with respect to the general public but not entirely so in the industry. My focus in this writting will be in the backend part of Engineeering.

A programmer is someone who writes code, a Software Engineer is someone who has the knowledge to oversee the production of Software from beginning to end keeping in mind principles like scalability, reliability, performance, architecture and so much more. Programmers are more intrested in codes, not concepts. If the code works, then everything is okay. They're the people who spend time online arguing about React Native and Flutter, NodeJs and Django etc.

You can't blame them, code is all they care about. They're the people wo believe Firebase is good for everything. They don't care about which database-type is good for the current project, as long as it can store data, they're good. After building an app that works well, when the App reaches 50,000 users and the server is no longer responding, the only thing they'll verify is if their code suddenly developed a problem. That's because they haven't studied concept like scalablity. They don't understand server overload, they don't understand load balancing, they don't understand database replication, they don't understand sharding, they don't understand the role of a Message Queue, they don't understand the role of CDNs and so much more. This is because all these principles are Software Engineering concepts which are used to keep Software alive and performance no matter the traffic. It's beyond coding. This is real Engineering.

Programmers don't like reading books

They're more intrested in Youtube videos which teach them how to clone a big website or app. Every good software engineer reads. Concepts are well explained in books. Read books like "System Design interview" by Alex Xu, "Designing Data-intensive Application" by Martin Kleppmann. Read books on software architecture, system design and many more. Learn enough to become a consultant. What happens when your number of users grow in size and one server cannot handle them, do you know what to do? Do you know the concepts to apply to keep the app running again? You can't store 1 million users on a single database. What happens if you are querying the data of the 1 millionth user? That'll make things terribly slow. So do you know what to do mitigate that issue? Even after sharding the database, how will you handle the shards of celebrities? Do you know what to do to handle traffic? Your code is perfect, there's probably nothing to change there, but now you need to use Engineering knowledge to solve that problem.
